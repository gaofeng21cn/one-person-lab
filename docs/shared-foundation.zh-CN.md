# 共享基础结构

`OPL` 把不同实验室任务组织进同一个体系的依据，定义为一组共享基础结构，而不只是它们都可以让 Agent 参与。

## 资产层

`资产层` 负责承载被多个任务反复消费的对象。

典型对象包括：

- 数据资产
- 文献与参考资料
- 图表与模板
- 已完成的正式交付物

如果没有这一层，论文、基金、学位论文和答辩材料就会各自复制一份上下文，最终失去一致性。

## 记忆层

`记忆层` 负责承载不适合只留在即时会话里的长期认知。

典型对象包括：

- 选题记忆
- 数据问题映射
- 期刊偏好
- 基金方偏好
- 评审经验
- 教学素材框架

这一层的目标不是保存所有对话，而是保存后续任务仍然会复用的结构化判断。

## 治理层

`治理层` 负责回答“什么情况下允许继续做，什么情况下应该停下来”。

它关注的是：

- 何时允许进入正式执行
- 何时需要补证据
- 何时应当止损或改题
- 何时可以进入正式交付

如果没有这一层，连续性与可审核性都会变弱，因为执行过程将失去明确的继续与停止条件。

## 交付层

`交付层` 负责把过程性产物收束为正式输出。

它关注的是：

- 哪些文件构成交付包
- 人类应该审核哪些表面
- 上游研究资产如何同步到下游正式材料

这一层对论文、基金、学位论文、幻灯片和其他视觉交付物都成立，只是具体交付协议不同。

## 智能体执行层

`智能体执行层` 负责让 Agent 真正成为可控执行者，而不是自由漂移的聊天接口。

它关注的是：

- 稳定入口
- 控制入口或任务路由
- 运行监控
- 审计落盘

这一层不是为了把人排除掉，而是为了让人只需要审核关键结果，而不必盯住底层执行细节。

## 当前已经比较清楚的部分

在当前体系里，最先成形的是 `Research Ops` 对这五层的具象化，也就是：

- `MedAutoScience` 已经在数据资产、运行治理、交付同步和智能体接入上形成了明确边界

同时，第二个正在浮现的具象化已经开始出现：

- `RedCube AI` 正在把视觉交付物的交付层与智能体执行层写实出来，当前最直接承接的是 `Presentation Ops` 的 `ppt_deck` family

这也是为什么 `OPL` 当前并不停留在概念层面，而是已经有一个已成形起点和一个正在收敛的第二实现面。
