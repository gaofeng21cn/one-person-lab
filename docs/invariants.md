# OPL 硬约束

Owner: `One Person Lab`
Purpose: `invariants`
State: `active_truth`
Machine boundary: 本文是核心人读真相面。机器真相继续归 contracts、source、CLI/API 行为、runtime ledger、provider receipt、domain-owned manifest 和真实 workspace / App evidence。

## 顶层定位

- 用户公共生命周期 owner 永远只有 `OPL Base`、`OPL App`、`OPL Packages`。内部 provider / adapter id 不得成为 CLI selector 或第四类 lifecycle owner；companion tools 只能是 Base dependency/integration status，Codex surface、managed runtime source、managed policy 与 workflow profile 都只能作为 Packages transaction 的 carrier/projection/profile migration。`opl update` 只管理 Base，`opl packages` 只复用 canonical package core；标准 Agent install/update/repair/rollback/uninstall 必须在同一闭包事务中管理 manifest 声明的 runtime source carrier，锁定 tree/source digest，并以当前可执行的 bootstrap、health 和 handler probe 作为 launch readiness，失败补偿不得删除 preexisting adopted source。runtime source 物理 mutation 必须先持久化 recovery marker，以 package lock/receipt 为 commit point并支持 crash 后幂等 rollback/finalize；commit 后 cleanup failure 只能是可重试 `cleanup_pending`，不得返回 unknown failure。derived runtime env 必须在 OPL state root，不能污染 source tree。任何 profile migration 必须 semantic merge fail-closed，任何 managed policy migration 必须 fresh typed inventory、backup、single receipt 与 LKG rollback，任何 package mutation必须保留 digest、dirty/developer checkout 保护和单 receipt。不得维护 workflow/package 专属第二账本。
- required capability dependency 必须按 closure transaction 安装、更新、repair 和 rollback；consumer/provider version、manifest/content digest、ABI、required Skill export 与 required module contract 必须进入 lock/readback。被 required consumer 引用的 provider 不能单独 disable/uninstall。scope activation 必须从 provider manifest 动态物化当前发布包声明的全部 exported Skills；MAS Scholar Skills 当前默认 scope 是 35 Skills，11 core + 8 modules 只是 readiness floor，module contract id 不得误物化为 Skill 目录。managed projection 缺失或漂移应在下一 use boundary 自动恢复；package 未安装或 closure 的 ABI/SemVer/trust/content-lock 不兼容一律阻断普通 launch，只开放 status/doctor/repair 或 typed update route。
- OPL Base 是 Framework、Temporal-backed provider runtime、共享 toolchain 与系统依赖的唯一 lifecycle owner。OPL App 只拥有自己的 GUI carrier/host update route；OPL Packages 只拥有 Agent/capability/workflow package closure、projection 与 receipt。标准 Agent 只能消费 Base，不得在 agent manifest、repo helper 或 package lifecycle 中安装、更新、回滚或反向管理 Base/Temporal。
- Homebrew 只承载 OPL Base，唯一 Formula 名是 `opl`，版本/source projection 来自 `framework_core.homebrew_formula`。任何 Agent、capability、workflow package 都没有 Formula/Cask。base-only 安装参数唯一为 `--skip-packages`；旧 `--skip-modules` 必须 fail closed，不保留 alias。
- first-party package canonical ids 固定为 `mas`、`mag`、`rca`、`oma`、`obf`、`mas-scholar-skills`、`opl-flow`。每个 canonical id 只能有一个 OCI repository `ghcr.io/<owner>/one-person-lab-packages/<canonical-id>`；旧 namespace 只作 migration/history locator。每个 Package 版本必须是 owner SemVer；moving channel 只能是 `candidate`、`latest-stable`；内容变化未推进 owner SemVer、同 version 漂移到新 digest、manifest/channel version 不一致、或 daily workflow 重发 unchanged component 必须 fail closed。Release Set generation 独立使用 `YY.M.D[-rN]`；`opl_release_set.v2` 必须记录 OPL Base、OPL App 与七个 Packages 的精确版本、source commit、artifact ref 与 digest。组件更新只比较 `component_id + version + artifact_digest`，`rN` 或仅 source commit 变化不得触发未变化组件重建或重装。每次 Package candidate build 必须先冻结 canonical seven 的 exact owner cohort lock，检测与发布只能消费同一 lock，普通 consumer 必须从同一 Release Set catalog selection 锁定 version、owner commit、artifact/channel digest，不能再次读取 moving tag。OPL Base archive 只允许 package runtime allowlist，必须包含 `bin/opl`、`dist`、contracts 与生产依赖声明，不得携带 `src`、`docs`、`tests` 或 workflows；非 runtime 改动不得改变 Base content identity。`one-person-lab-manifest` 只可作为 Release Set catalog carrier，不能成为第八个 Package identity、依赖节点或用户 lifecycle owner；candidate 与 stable 必须由唯一 Framework workflow写入，App 只能以 request id 驱动 candidate -> latest-stable fail-closed saga并消费 machine receipt，stable promotion 只能 retag 已验证的 immutable generation digest。Base、App、七个 Packages 与 catalog 必须有 exact-byte SBOM/provenance attestation、OCI digest readback 和 per-component OCI referrer；缺任一项不得返回 promotion success receipt。
- `OPL` 是面向高价值知识工作的完整智能体运行框架。它以 `Codex-default session runtime` 为默认交互底座，以 stage-led family framework 承接长期自治、恢复、provider-backed stage attempt、human gate、trace、projection 与交付收口。
- `OPL` 的默认交互与具体执行 runtime 是 `Codex CLI`；`Codex CLI` 是阶段内默认最小执行单元。Full online family runtime 的 readiness 对象是已配置且 ready 的 Temporal-backed family runtime provider。Temporal 是 OPL production online runtime 的必需 substrate；`local_sqlite` 只作为已退役 provider 词汇和负向 guard，SQLite sidecar 只承担 projection/readback index，不能替代 Full online readiness；`hermes_agent`、`claude_code` 与 `antigravity_cli` 只允许作为显式非默认 executor adapter/backend，Hermes provider / Gateway / readiness / compat 面只允许作为历史 provenance、诊断语料或负向 guard。
- `Codex CLI` 是 OPL 的受管 runtime dependency：OPL 必须检测实际命中的 binary、版本、最低版本策略和 PATH 候选；同版本兼容 wrapper / alias 归并到当前有效入口，低于当前最低版本或当前命中版本无法解析的 Codex CLI 只能进入 `attention_needed`，不得被报告为 ready。
- `Codex CLI` stage runner 必须保持 progress-first closeout：typed closeout 是 owner/quality/ready claim 的高质量载体，不是 stage transition hard gate。free-text、raw output、部分 artifact、阴性/零结果或无输出诊断都必须被物化为 `completed_with_quality_debt` progress artifact，使 Codex 可选择 advance、skip、repeat、reverse 或 route-back；provider completion、测试通过或文件存在仍不能授权 domain/quality/ready claim。`--output-last-message` 可作为 typed closeout 捕获面；`--output-schema` 只能按 provider capability 启用，当前 `gflab` provider 不得携带 `--output-schema`。格式漂移只降低 closeout 质量与可观测性，不能变成第二套推进控制面。
- 只有显式 domain activation 或显式 runtime switch，才允许离开 Codex-default 语义。
- 大型任务必须按 stage 作为可观察、可恢复、可审计的工作单元推进；不得把开放式知识工作降级成只靠硬编码步骤或固定脚本后处理的流程。
- OPL 必须坚持 AI-first、executor-first、AI 原生专家判断优先、contract-light。智能体能力提升主要来自 `Codex CLI` 等 AI executor、domain stage pack、prompt、skills、tool affordances、knowledge、rubric 和 quality gate 的演进；合同只负责边界、安全、权限、凭据、可写范围、审计、receipt、阻塞、恢复和 projection 这些下限，不负责把开放式智能行为、工具编排、创作策略、评审方法、路线判断或修订逻辑写死。
- AI 原生专家判断优先级高于机械分数、checklist、schema 完整性、contract completeness、descriptor readiness、provider completion、generated-surface proof 或 App/read-model proof。上述信号只能作为 advisory、blocker localization、evidence gap 或 route-back 输入；除非独立 AI stage、domain-owned quality gate、owner receipt 或 typed blocker 关闭对应判断，否则不得被写成 domain ready、quality verdict、artifact ready、publication/fundability/visual/export ready 或 production ready。
- 涉及知识交付、专家判断或正式交付质量的复杂步骤必须是 first-class stage，例如 AI 审稿、publication quality review、fundability review、visual review、revision/rebuttal review；不得把这类流程作为另一个 stage 内部的普通函数、helper、后处理或 authority function 暗中完成。
- 每个标准 OPL Agent stage 必须声明本 stage 的 prompt、skills、tool affordance boundary、knowledge 和 quality gate。skills 是 Codex / domain skill 或 repo-owned operating guidance 的显式 refs；tool affordance boundary 只声明可用能力、权限、凭据边界、可写范围、side effect 风险和 forbidden authority，不声明工具使用顺序或认知流程；knowledge 可以包含 source refs、memory refs、经验卡、rubric、policy 和 prior receipt refs；quality gate 必须说明本 stage 如何被独立审核、何时 route-back、何时允许 handoff。
- Stage admission 只保留一套最小运行边界。`launch admission` 仅能因 wrong-target identity/currentness、selected executor 不可用、权限/凭据/安全、authority/forbidden write、不可逆动作授权或明确 human decision 硬停；scope、role artifact、manifest、packet、receipt、review、prompt/skill/tool/knowledge/rubric refs、lineage 或 conformance 缺口全部成为 advisory / quality debt context。`closeout admission` 对每次已发生的 attempt 都必须接受 domain artifact、raw output、阴性/零结果或 no-output diagnostic 作为 progress closeout；required role、manifest、owner receipt、content hash 和 generation 只决定能否声明 domain/quality/export/ready，不决定下一 stage 能否启动。
- 每个标准 OPL Agent stage 应声明 `domain_stage_completion_policy`：stage 内由 domain stage owner 判断内容质量，并输出 next / route-back / stop / human-gate 建议；Codex 是 stage 语义路由 owner，OPL 只记录 StageRun / attempt、运输 refs 和投影 next owner delta。`provider completion is domain completion`、`file presence closes quality gate`、`suite pass closes stage quality`、`conformance pass closes stage quality` 或 `OPL judges content completion` 都是 forbidden authority。缺 completion policy、typed packet 或 accepted owner-answer shape只形成 conformance quality debt；运行时必须生成 progress diagnostic，不得阻断后续 stage。
- `Evidence-Grounded Decision Agent Profile` 在无 evidence refs、低置信、冲突证据未解释或 required tool/provider receipt 缺失时，应保留现有证据与 diagnostic，关闭 success/ready claim，并由 Codex route-back 或推进到 reviewer stage；只有 sensitive external egress、权限/安全、authority、不可逆动作或明确 human decision 才硬停。Profile 产出的 `decision-support artifact + evidence trace` 只是 owner 决策输入，不是 domain verdict、final decision、quality/export verdict、artifact authority、owner receipt、typed blocker 或 human gate；OPL 只能检查 flow refs、shape、authority boundary 和 evidence coverage，不能替 domain / App / human owner 签最终答案。
- `opl profiles select|inspect|conformance` 是 OPL-owned refs-only capability selection / structural conformance surface。Profile catalog、standard scaffold 和 refs-only template 只能作为 lower-bound conformance guardrail、physical skeleton 与 shape ref，不是 target-agent design source，也不能限制目标 agent 设计上限。OMA 和其他 agent builder 必须消费它来确定基座 profile、source-derived design route、requirements 和 conformance gaps。非英文或混合语言 intent 可用重复 `--intent-signal` 传入 catalog canonical trigger；signal 不得替代原始 intent、reference design 或设计提炼。内置 catalog profile 只能作为 lower-bound guardrail；当无内置匹配但有论文/PDF/repo/产品案例等 reference design source 或 pattern packet refs 时，selector 必须返回 `profile_selection_mode=source_derived_design`，并声明 `ReferenceDesignPacket`、`TransferMap`、`AgentPackPlan` 三类设计消费对象或等价 machine fields，由 OMA 提炼设计思路并组装目标 agent pack。conformance 不能只因为看见 source-derived `route_ref`、object ref 或 requirements 通过；它必须读取实际 typed objects，逐 workflow step 核对 provenance/source anchors、TransferMap mapping、AgentPackPlan 独立 stage、stage control plane 真实 stage 与 DesignAdmissionReceipt 准入关系，空心对象、缺 mapping、stage graph collapse、plan-only graph、未物化或未准入 stage 必须 fail closed。`AgentBuildReceipt` / `build_receipt` 是物化后的构建证明，不是第四个设计对象。缺内置 profile 且缺 source-derived design refs 时才 fail closed。该 surface 不能写 target domain truth、不能替 OMA 生成目标领域知识正文、不能签 target owner receipt、不能声明 target agent delivered / domain ready / production ready。
- source-derived conformance 必须覆盖 `AgentPackPlan.planned_stage_refs` 的完整集合，而不只覆盖论文 pattern stage：`source_pattern_ref` 与 `target_only_requirement` stage 都必须在 stage control plane 中恰好物化一次，origin 与准入 receipt 必须一致，未计划 stage、重复 stage、漏 stage 或 target-only requirement 未准入都必须 fail closed。正式 `AgentBuildReceipt` 只能在文件物化完成后生成，必须声明 `receipt_timing=post_materialization`、完整 materialized stage ids、planned-file presence proof 和每个计划文件的 SHA-256；预物化 draft 只能保留 expected receipt ref，不能伪造 receipt body。
- AgentBuildReceipt 的 digest target 解析、JSON Pointer 选择和 JSON normalization 必须由 OPL `reference-build-proof` primitive 持有并被 OMA 消费。对嵌入 receipt 的 JSON control/capability 文件，canonical digest 必须递归移除 `build_receipt / build_receipt_ref / build_receipt_refs` 后再序列化；不得改成跳过这些文件、信任 receipt 自报 digest、或由 OMA/OPL 分别维护两套算法。
- `opl profiles capability-plan` 是同一 profile front door 的 refs-only capability planning 子面。它只能消费 canonical selection receipt、Framework-owned 固定 `profile_capability_plan_input`、显式 exact capability refs、显式 owner repo 的 `contracts/capability_map.json` 与既有 Connect exact resolver；不得扫描 HOME、维护 central specialty catalog、复制 OMA object ABI 或 owner module body、激活 legacy id、做 heuristic score / semantic equivalence，或执行 search/sync/install/cache/download。dependency / environment / Pack OS descriptor / Pack lock 只能输出带 owner root 与 source fingerprint 的条件化 action refs；optional miss fail open，只有绑定 canonical current-owner-delta 的 route-required hard-boundary exact miss 可生成 blocker candidate，且 OPL 仍不能创建 typed blocker instance、写 owner receipt 或声明 dependency / environment / target / domain / production ready。
- StageRun 只运输 Codex 选择的 stage、attempt lifecycle、workspace 与可读 artifact refs，不设 execution authorization、stage-packet 格式门或 closeout-binding 推进门。缺 manifest、packet、role、receipt、review、capability binding、provider attempt 或 lease 只降低可观测性与质量声明，不阻止 Codex 从 declared stage、主提示词、workspace context 和已有可读产物启动下一 stage。Currentness identity 缺失只禁用 stale reuse；只有 identity 冲突可能写错目标、权限/安全/authority、不可逆动作或明确 human decision 才能硬停。
- Stage Strategy Kernel 是 stage 内的认知计算内核，负责组织 candidate generation、reflection / review、ranking / selection、evolution / revision、strategy retrospective 这类开放式策略循环。它只能以 prompt / skills / tool affordance boundary / knowledge / rubric / quality-gate refs 的形式由 Foundry Agent stage pack 显式声明，并由 selected Agent executor 执行；不得被实现成 OPL 硬编码 workflow engine、工具编排器、机械 checklist、score reducer、domain truth store、artifact authority 或 quality verdict owner。
- Stage Attempt Runtime 只负责可恢复 attempt：identity、idempotency、provider binding、workspace/runtime root、heartbeat、checkpoint、resume、retry/dead-letter、human gate、closeout refs、usage/timeline 和 projection。它不得替 Stage Strategy Kernel 生成、评审、排序、修订或学习，也不得把 provider / executor completion 写成 quality gate passed、owner receipt observed 或 stage progression ready。
- StructuredCloseoutGate 是 OPL Framework / Runway 的 claim-validation primitive，不是 MAS 私有兜底或 stage 外控制层。合法 typed JSON closeout packet / refs 可以授权 owner/quality/ready claim；`Codex CLI` prose、raw output、provider completed、tests green、file presence、docs patch或 read-model refreshed只能关闭为 progress / `completed_with_quality_debt`。格式漂移按 terminal capture、session recovery、same-session enforcement、domain receipt recovery、raw-output/no-output diagnostic 的顺序归一化并继续推进，不得生成 provider-runtime blocker。Repair / redrive 只能作为 Codex query / decision context，不能伪造 domain owner receipt、typed blocker 或 human gate，不能写 domain truth、artifact body、memory body、quality verdict、runtime ready、domain ready 或 production ready。
- Stage 间的 `requires` / `ensures`、显式 refs、human gate decision 或 owner receipt 只是 Codex 的输入与 claim evidence，不是程序 transition gate。OPL 不得用启发式伪造缺失证据，也不得把 provider / executor 完成状态写成 quality/ready 条件已满足；缺失前置输入必须显式进入 quality debt，下一 stage 仍可启动并自行判断继续、reverse 或 route-back。
- `Stage Transition Authority`、transition intent/event ledger 与 accept/reject decision 已退役，不得复活为持久化或语义 primitive。Codex CLI 是唯一 stage route selection owner，可选择 advance、skip、repeat、reverse 或 route-back；domain agent、reviewer、human gate 和 provider只提供 artifact、diagnostic、owner answer与 route recommendation。OPL 只能记录 Codex 已选择的 stage attempt 与 transport observation，不得以 schema、admission、readiness、counter、provider completion、conformance、App projection或 operator read model拒绝、覆盖或自动改写 Codex 路由。
- stage pack 可把 source/artifact/workspace scope 做成 refs：`source_scope_refs`、`artifact_scope_refs`、`workspace_scope_refs` 只允许 OPL 投影和检查可见性，不授予 OPL source truth、artifact authority 或 workspace truth。缺失、过期或普通冲突的 scope进入 quality-debt diagnostic 并由 Codex选择继续或 route-back；只有 identity/currentness 冲突可能写错目标、权限/安全/authority或明确 human decision 才硬停。
- 可启动 stage pack 的 `runtime_assumptions` 应能被 `monitor_refs` 或 assumption-local monitor refs 支撑；stale assumption、缺 monitor ref 或缺 owner 默认进入 readiness warning 和 minimal counterexample，不能被写成 App/operator ready。只有当缺口同时影响启动安全、越权、边界事件记录或 replay/audit 基础证据时，才升级为 typed blocker / human gate / route-back。
- Stage pack hash 变化必须经过显式 migration policy；已有 attempt 只能继续绑定旧 hash、迁移到新 hash，或被 human gate 阻断，不能静默切换可复用 stage pack。
- 标准 OPL Agent 的可持久化 stage attempt 应按 `Stage Folder + progress artifact/diagnostic + optional stage_manifest/role artifacts + owner receipt/typed blocker + minimal StageRun state` 物化到外部 runtime artifact root。StageRun Kernel 只是最小状态壳，承载 generation、attempt、lease、retry/dead-letter、event log、current owner delta 和 projection rebuild，不是第二 controller、readiness worklist、domain scheduler 或 quality verdict owner。stage artifact 的 `latest` / `current.json` 只能表示 artifact attempt pointer / current projection，不能写 StageRun current pointer、StageRun terminal state 或 `current_owner_delta`。缺 owner receipt、manifest或 hash 的文件仍可作为下一 stage 的非权威 input；它们只关闭 quality/export/ready claim。receipt 指向文件缺失或 hash 不匹配同样形成 broken-artifact diagnostic，不阻断 Codex推进。
- Workspace topology 的机器边界必须保持 `Workspace Group -> Project Unit -> Stage Artifact Unit -> Owner Receipt / Typed Blocker`。Project Unit 的默认 physical collection 固定为 `projects`，新 workspace 默认生成 `projects/<project-id>`；`studies` 与 `deliverables` 只能作为 display label、legacy alias、adopt/provenance terminology 或 domain semantic alias，不得重新成为 canonical project root。Project Unit 的 Stage Native 默认 stage-owned artifact root 是 `artifacts/stage_outputs`；ordinary user / App 默认检查面必须先指向 workspace-local stage outputs、domain-owned product views 和 owner next action。`workspace_inspection.json`、`workspace_resource_inventory.json`、project-local `stage_outputs_index.json` 和 `current_stage.json` 只能作为 OPL-owned refs-only projection：它们可以帮助用户检查路径、资源 inventory、stage lifecycle shape 和当前 pointer，但不得成为 domain truth、stage closeout truth、owner receipt authority、typed blocker authority、quality/export verdict 或 production readiness 证据。`${OPL_STATE_DIR}`、`runtime-state/`、provider history、SQLite sidecar 或 runtime backing directory 只能作为 execution backing、refs-only index、restore/provenance 和 audit drilldown，不得成为 canonical project root、普通用户默认查看面、stage closeout truth、owner receipt authority 或 typed blocker authority。
- Workspace governance v2 的 generated truth 必须写入 workspace-local `control/opl/projections` 和 `control/opl/reports`：`control/opl/projections/{workspace_map,workspace_health,workspace_inspection,workspace_resource_inventory}.json` 是 canonical projection root，`control/opl/reports/workspace_report.json` 是默认用户检查摘要；workspace 根层 `workspace_map.json`、`workspace_health.json`、`workspace_inspection.json`、`workspace_resource_inventory.json` 和 `workspace_report.json` 只能作为兼容 mirror。`workspace validate` / `doctor` 必须按 Contract Light 诊断分级：缺 workspace path/root/index、index shape、agent/profile identity、project collection/root、stage index/current pointer 形状、authority overclaim 或 runtime-state overclaim 才能作为 hard blocker 阻断默认执行；canonical generated projection missing/drift、root mirror drift、`workspace_report` missing/drift、profile binding missing/drift、topology event missing、`agent_workspace_norm` projection drift、shared manifest 和可重建 stage projection 缺口都必须进入 repairable finding，并指向 `opl workspace upgrade --workspace <path> --apply`，不得把小投影漂移升级成默认运行阻断。只匹配 norm id/version、只存在 root mirror、或只通过 conformance summary 都不能证明 workspace governance current。
- 每个 workspace instance 必须记录 profile binding：`profile_version=workspace-topology-profile.v2`、`profile_fingerprint=opl-workspace-topology-profile-v2-projects-stage-outputs`、profile contract ref 和 migration history。`topology_events[]` 是 topology mutation 的审计面；初始化、adopt、upgrade、project lifecycle mutation 或 generated projection refresh 不能只靠文件时间或 registry binding 推断。
- Project lifecycle 的唯一通用状态集合是 `active`、`paused`、`archived`、`superseded`、`locked`。这些状态只表示 OPL workspace projection lifecycle，不关闭 stage、不删除 project root、不签 domain receipt，也不改变 domain truth。physical delete 必须有 domain owner receipt；domain repo 可以声明 locator、domain semantic label、owner receipt / typed blocker 和 product view，但不得持有 generic workspace lifecycle 或把自己的 `study` / `deliverable` lifecycle 升级成 OPL Project Unit lifecycle。
- `workspace list` 必须保持 registry-only catalog；registry-wide 结构审计属于 sibling surface `workspace fleet report`。Fleet report 只能读取 registry binding、workspace doctor/report 和 refs-only counts，不执行 direct-entry command、manifest command 或 domain product-entry resolver，不读取/复制 domain manifest body，也不得声明 domain ready、stage complete、artifact ready、quality/export ready、production ready 或 L5。shared resource manifest 的 `resources[]` 只能保存 refs-only provenance / checksum / reuse / staleness 记录，`body_ref` 必须为空；`workspace upgrade` 必须保留这些 records，`workspace inventory` 只能投影 record refs 与计数。

- OPL family 的目标态高于当前实现分布。MAS/MAG/RCA/OMA/OBF 当前存在的私有 scheduler、runner、session store、SQLite/lifecycle、workspace/source intake、memory/artifact transport、workbench、sidecar/status/product wrapper 或 generated wrapper，只能作为迁移输入；不得因为已有 active caller 就写成长期合理。

- 当 OPL primitive、pack compiler 或 App shell 还不够成熟时，应在 OPL 层定义缺口、必要时调研外部成熟系统，并把结论沉淀为 OPL generic primitive / generated surface / policy；不得让 MAS/MAG/RCA/OMA/OBF 各自复制私有平台。
- 文档和计划必须先设理想态，再找差距；差距不是妥协清单。为了理想态，可以做革命式重构并完全抛弃旧模块、旧接口、旧测试、旧目录和旧文案；处理清楚 active caller、替代 surface、provenance 和必要证据后，不保留历史兼容面。

## OPL 内部 8 项原则

这 8 项原则是核心五件套、active gap、closure matrix、contracts README 与用户级 `~/.codex/TASTE.md` 的共同读法。公开白皮书保留 5 条高层哲学；内部读面用这 8 条承接原十项机器边界、contract-light 语义和长期维护偏好。

1. **真相归主**：领域事实、质量裁决、产物权威、owner receipt 和 typed blocker 留在对应 owner。OPL 只持有 framework-level runtime / activation / discovery / projection；App 只持有产品入口、GUI product truth、release gate 和用户投影；read model、generated surface、provider proof 或 cleanup proof 不能签发 domain ready、artifact ready、quality verdict 或 production ready。
2. **抓大放小**：hard blocker 只覆盖启动安全、越权、关键 runtime event 记录缺失、composition 不满足、hard human gate、executor binding、不可逆权限和负责人回执。capacity、monitor、assumption、cohort-loop、replay、domain-owner review、diagnostic proof、L5 evidence 细项和 provider ops detail 默认进入 advisory refs、audit、cleanup 或 evidence lane；只有破坏大边界时才升级。
3. **目标先于路径**：先按用户目标、owner decision、human gate、artifact delta 或可接力结果定义进度，再选择 runtime、stage、docs、测试和修复路径。OPL substrate 是路径不是目标；除越权、缺少必要输入或同一写集 ownership 冲突外，platform repair 不能吞掉目标主线。
4. **AI-first, contract-light**：开放式专家工作优先交给 AI executor；`Codex CLI` 是默认 selected executor，`hermes_agent`、`claude_code`、`antigravity_cli` 或其他 adapter 只能通过 request、stage attempt 或 handoff 显式绑定，并以 `executor_binding_ref`、receipt / audit / fail-closed 证明连接。合同只绑定 prompt、skill、tool affordance boundary、knowledge refs、expected receipt、authority boundary 和恢复/投影下限；工具 refs 是 affordance catalog，不是 workflow script。
5. **阶段交付，交付即推进**：Stage pack 是启动单位；OPL 准入并启动带目标、输入、输出、owner、权限、质量门和回执形态的 stage，不启动自由形态 workflow script。`requires` / `ensures` 组合在启动前检查；domain judgment 仍是 runtime / domain-owned 结果。阶段形成可接力成果、owner receipt、typed blocker、human gate、route-back 或明确停止决策后，就进入下一步。
6. **单源派生，薄入口**：关键动作、阶段、合同、路由、状态和描述符从 canonical source 派生到 CLI、API、App、Skill/plugin、MCP、文档和报表。入口稳定而薄，复杂逻辑进入清楚模块；`verified_static_core` 只覆盖 identity、owner、refs、scope、composition 与 forbidden-authority 约束。
7. **结构收敛，历史退役**：当前责任面替代旧入口后，旧模块、alias、facade、wrapper、兼容测试和过时文档应删除、归档或 tombstone。历史只保留 provenance，不继续污染当前路径；旧 gateway、federation、frontdoor 等路线只能在 history / tombstone 语境中出现。
8. **证据匹配风险**：文档和结构整理用最小充分验证；权限、artifact mutation、memory writeback、人类决策、外部系统、domain verdict、release/currentness/readiness claim 必须用对应 owner receipt、live/readback、runtime artifact 或端到端证据。`runtime_enforced_boundary` 覆盖 AI 输出、人类决策、外部系统、artifact mutation、memory writeback 与 domain verdict。descriptor ready、read model 可读、generated-surface proof、provider proof 或 cleanup proof 都不等于 domain ready、artifact ready 或 production evidence complete；普通未闭合质量边界记录 quality debt / review / route-back recommendation 并继续，只有真实硬停才返回 typed blocker、human gate 或 receipt conflict。

- Gap 文档是当前 active work tracker，不是完成史。功能/结构 gap 落地后，active gap 文档必须删除或重写已关闭条目；当前没有 gap 时保持薄 no-gap baton。已完成的调研、规划、closeout、branch/worktree、receipt 流水和验证过程只能进入 `docs/history/**`、runtime ledger、owner repo provenance 或提交历史。

## Stage 内正式 Review

- 同一目标的多次生成、审阅和修复是 Attempt；目标、owner、质量门或交付关系发生变化，就是新的 Stage。
- Attempt role 只能是 `producer | reviewer | repairer | re_reviewer`。Attempt 不得拥有 `requires`、`ensures`、next-stage refs、route、sub-stage graph、独立 owner、Stage current pointer 或 Stage transition authority。
- 跨 Stage 语义路由只能来自 StageRun 的终局 Codex Attempt：primary-only Stage 是 producer；启用正式 Review 的 Stage 是 terminal reviewer / re-reviewer。repairer 不得做终局 route selection，repair-required Review 与其他非终局 Attempt 只能返回有证据的 route recommendation；runtime closeout guard 必须拒绝它们写入终局 route fields，并拒绝 decision/recommendation 并存、legacy route 字段、非法 shape、undeclared target、伪造 controller metadata 与无效 finding-closure。StageRun controller 只编排 child Attempt；OPL transport 只物化通过 ABI/authority 校验的 Codex-selected declared Stage，不判断该专业路线是否正确。无效 route output 不得丢弃可消费 artifact，只形成 route quality debt；顺序 fallback 只能来自 domain pack 已声明的默认路径，不能由 Framework临时推断。
- 只有非模型 Temporal `StageRunWorkflow` controller 可以创建下一权威 Attempt；Codex executor 和 Codex subagent 都不得扩张质量循环拓扑。
- StageRun 只能由 `opl family-runtime attempt create` 消费已编译的 `opl_pack_bound_stage_quality_runtime_binding` 创建；binding 必须绑定 Stage manifest ref/SHA、quality policy、四类 role prompt、rubric、goal/source/lineage refs。raw `family-runtime stage-run start` 永久退役，`stage-run` CLI 只允许 query；不得恢复 unbound StageRun 创建入口。
- 正式 Stage Review、Repair 和 Re-review 必须各自使用新的 `StageAttemptWorkflow`、新的 `codex exec` 与不同 thread/session，不继承 producer/reviewer conversation history。same-thread refinement 与 typed-closeout resume 都不算 Review。
- producer / repairer 的 reviewable artifact 必须由 domain-owned closeout ref metadata 提供 exact ref 与 SHA-256 identity；review receipt 必须绑定该 identity、producer/reviewer Attempt 与 session、rubric、`no_context_inheritance=true` 和 verdict。文件存在、provider completion、OPL projection 或 hash 推测均不得生成 review receipt。
- 一个质量循环最多包含一个 producer、一个 initial reviewer 和三组 repairer/re-reviewer。initial review 不计修复轮次；provider/activity/structured-output/runtime retry 不得复用质量预算字段或计数。
- Re-review 必须按稳定 finding id 关闭 initial findings；只有 required finding 未关闭、repair regression 或 critical new finding可以继续修复。普通新建议只能形成 optional observation / quality debt。
- 三轮耗尽且仍有可消费产物时必须终止为 `completed_with_quality_debt` 并继续 Stage 图，但阻止 quality/export/publication/submission/ready claim；无可消费产物或真实 authority/safety/human/currentness 硬门才可 blocked / human gate。
- Meta Review 是独立 StageRun，必须上下文隔离；它的主 Attempt role仍为 `producer`，不得新增 `meta_reviewer` 等自定义 Attempt role，也不得递归启动 Stage 内正式 Review。
- Packaging/Handoff Stage 必须在 manifest 声明最终产物风险。若它在上游 Review 后生成或转换需要专业判断的新交付 bytes、冻结 canonical bytes，或首次签发 quality/export/publication/ready claim，必须启动 fresh formal Review；若只运输已经审阅的 immutable refs，或仅对已审 bytes 做确定性机械封装，且 acceptance 明确保留给下游 owner，则可以只有 primary Attempt。human gate 与 formal Review 正交，不能互相替代或互相推导。
- 普通产品投影不得把 Attempt 展示为 Stage。Attempt、thread/session、artifact identity receipt、review receipt、finding/repair lineage与成本只属于 developer/operator drilldown。
