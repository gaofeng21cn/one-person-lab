## RCA Owner 边界

RCA 必须继续持有：

- source readiness、communication strategy、storyline、visual direction 和 route decision；
- `ppt_deck`、`xiaohongshu`、`poster_onepager` 等 deliverable family 的 stage pack、prompt / skill、route policy 和 review gate；
- visual director review、screenshot review、repair decision、export proof interpretation、ready / exportable / handoffable verdict；
- canonical artifact refs、artifact mutation permission、export bundle authority 和 handoff packet meaning；
- visual pattern memory body、route caveat、writeback proposal、accept/reject decision 和 receipt；
- domain owner receipt、typed blocker、no-regression evidence 的 RCA 语义；
- direct app skill path 与 OPL-hosted path 共用的 downstream service-safe domain entry。

OPL / App 可以提供：

- provider-backed stage attempt、queue、wakeup、resume、retry/dead-letter、human gate 和 attempt ledger；
- workspace/source intake shell、artifact locator shell、gallery shell、handoff shell、restore/retention ledger 和 operator projection；
- route/decision graph renderer、review/repair queue transport、proof locator、freshness、attention queue 和 repair command projection；
- native helper catalog / execution envelope、provider receipt、SLO / stale scan / observability drilldown；
- memory locator / index / writeback transport、proposal refs、accepted/rejected receipt refs 和 body-free grouping；
- App/workbench 的通用展示和 action routing。

OPL / App 不能做：

- 选择 RCA visual route；
- 生成或改写 RCA canonical artifacts；
- 下 RCA review/export verdict；
- 写 RCA visual pattern memory body；
- 把 provider completion、descriptor aligned 或 no-regression ref 升级成 visual ready / exportable / handoffable。

## 完善计划

### P0：冻结 RCA-only 计划边界

目标：让这份文档只服务 RCA，不成为 MAS/MAG/MDS/App 的跨仓 backlog。

动作：

1. 保持本文为 RCA active support plan。
2. RCA 当前事实继续以 RCA status / project / architecture / contracts / manifest 为准。
3. OPL 相关内容只写 owner split、托管接口、projection/read-model 责任和验收边界。
4. MAG/MAS 只作为“类似 owner split 的参考样板”出现，不列它们的实施 lane。

验收：

- 本文不再包含 MAS/MAG/MDS/App 的独立 repo gap matrix 或执行 lane。
- Active README 对本文的描述改为 RCA-only。

### P1：真实 RCA controlled visual-stage attempt

目标：把 RCA 从 descriptor/projection evidence 推进到真实 visual-stage owner receipt。

动作：

1. 选一个低风险真实 workspace，优先 `ppt_deck` 或 `xiaohongshu`。
2. 通过 OPL-hosted attempt 或 direct product-entry continuation 进入同一 RCA service-safe domain entry。
3. 覆盖至少一个完整 visual closure slice：source intake / plan refs -> visual direction -> artifact creation -> review -> repair or export。
4. RCA 返回三种之一：`domain_receipt`、`typed_blocker`、`no_regression_evidence`。
5. receipt 中明确 artifact refs、review refs、blocked item、repair target、export proof、residual risk 和 no-forbidden-write proof。

验收：

- OPL closeout 或 RCA session surface 能定位同一 owner receipt ref。
- Provider completion 与 RCA visual ready / exportable verdict 分轴显示。
- 真实 PNG/PPTX/PDF、receipt instance、memory body 不进入 repo source tree。

### P2：Artifact-producing owner receipt hardening

目标：让 RCA 每次成功、阻塞或回修都能给出可消费的 owner return shape。

动作：

1. 统一 `domain_receipt | typed_blocker | no_regression_evidence` 的 RCA 字段含义。
2. 将 artifact delta、review state、repair target、export proof、handoff packet 和 residual risk 纳入 receipt refs。
3. 让 direct path 与 OPL-hosted path 使用同一 return shape。
4. 对 provider timeout、review hard-block、export failure、source gap、memory apply denied 分别给 typed blocker。

验收：

- OPL/App 只需读取 receipt refs 和 typed blocker 即可展示下一步。
- RCA review/export gate 仍是唯一 ready/exportable authority。

### P3：Visual pattern memory live apply

目标：把 visual memory 从 locator/proof 推进到真实 writeback 闭环。

动作：

1. 从真实 closeout 提取可复用 visual lesson proposal。
2. RCA owner 执行 accept/reject。
3. accepted lesson body 写入 RCA-owned runtime/workspace memory store，不写入 OPL state 或 repo source。
4. OPL 只展示 consumed refs、proposal refs、accepted/rejected receipt refs、freshness 和 grouping。

验收：

- 至少一条 accepted 和一条 rejected receipt 可由 RCA surface 读取。
- OPL read model 继续 body-free。
- memory receipt 不自动产生 visual ready / exportable verdict。

### P4：Visual workspace lifecycle apply

目标：让 cleanup / restore / retention 从 contract proof 进入真实 workspace receipt。

动作：

1. 对真实 RCA visual workspace 执行 cleanup / restore / retention guarded apply。
2. RCA 返回 artifact mutation receipt 或 typed blocker。
3. OPL 只维护 locator、retention ledger、restore/provenance projection 和 operator action。
4. App 展示 artifact refs、freshness、mutation owner、restore proof 和 residual risk。

验收：

- artifact mutation 的授权来源是 RCA receipt。
- OPL/App 不删除、重写或生成 RCA canonical artifact。

### P5：OPL/App workbench projection

目标：把 RCA 用户可见进展做成通用工作台投影，但不把 App 做成 truth owner。

动作：

1. `Workspace`：展示 source readiness、material gaps、workspace locator、artifact root 和 recent activity。
2. `Deliverables`：展示 family、topic、deliverable、route、stage、progress、route decision refs。
3. `Review`：展示 blocked pages/items、review verdict refs、repair status、screenshot/export proof refs 和 residual risk。
4. `Artifacts`：展示 PPTX、PDF、PNG、publish copy、manifest、gallery、handoff packet refs。
5. `Attention`：按 owner/action 分组展示补资料、确认方向、批准继续、repair、provider wait、domain owner action 和 external publish/manual send。
6. `Safe actions`：按钮路由到 OPL provider signal、RCA product-entry、RCA sidecar 或 manual handoff，并返回 receipt/typed blocker。

验收：

- App 不产生 visual ready / exportable verdict。
- 用户能看清“素材是否够、路线为何这样选、哪页被挡住、谁负责、下一步能做什么”。

### P6：Legacy physical cleanup

目标：在 replacement proof 稳定后删除旧 active-path residue。

动作：

1. 扫描旧 Hermes / Gateway / local-manager / bridge wording 和 active caller。
2. 确认 direct skill path、OPL-hosted path、fixtures、tests 和 provenance refs 不再依赖旧入口。
3. 有 no-active-caller proof 的项删除或 tombstone。
4. 仍被合同引用的历史文档保留在 history/provenance 语境，并补 lifecycle note。

验收：

- active docs 不再把旧路径写成默认 owner。
- 删除不会破坏 direct path、OPL-hosted path 或 provenance tests。

## 从 MAG/MAS 类文档借鉴的模式

RCA 可以借鉴这些跨仓模式，但不在本文维护 sibling repo 的具体计划：

- `owner split first`：先写清 domain owner 和 OPL owner，再列动作。
- `authority pack`：domain repo 保留专业判断、quality verdict、memory body、artifact/export authority。
- `shared shell`：workspace/source intake、memory locator、artifact lifecycle、operator projection、SLO 和 App shell 上收到 OPL。
- `receipt over prose`：生产闭环以 owner receipt、typed blocker、no-regression evidence 和 workspace proof 为准。
- `body-free projection`：OPL/App 只展示 refs、freshness、owner、next action，不保存 domain memory body。
- `no-forbidden-write`：真实 artifact、receipt instance、memory body 和 export bundle 不进入 repo source tree。

这些模式可以帮助 RCA 规划自己的完善路径，但 MAS 的 paper-line、MAG 的 grant-stage、MDS 的 archive/oracle、App 的 GUI shell 都不属于本文的 RCA 实施项。
