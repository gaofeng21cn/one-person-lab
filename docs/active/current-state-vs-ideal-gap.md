## 文档读法

- `定位`：本文是 OPL family 的 active gap / implementation order，不是 north-star 目标态本身；目标态回到 `opl-family-agent-ideal-state.md` 和各 domain ideal-state 文档。
- `当前实态`：只把 fresh CLI、runtime ledger、代码审计和 domain-owned manifest 作为当前状态证据；不把未跑出的 long soak 或未迁移代码写成已完成。
- `差距定义`：功能/结构差距按目标态判断，不按现状是否可用判断。凡是 MAS/MAG/RCA 内仍承担通用 runtime、runner、session、lifecycle、workbench、memory/artifact transport、sidecar/status/product wrapper 或 generated surface 职责的实现，都应列为 OPL 上收、generated surface 替换、收薄或退役差距；不能因为它现在已有 caller 就写成长期合理。
- `最短路径`：功能上收、迁移、清理和 template 抽取先于大规模 integration/live soak；当前目标进一步收紧为 OPL-hosted `Declarative Domain Pack + minimal authority functions`。OPL 不只接收 domain thin adapter，而要提供 pack compiler / generated entry surface / generated sidecar / generated workbench / generic runtime primitives；MAS/MAG/RCA 默认只保留声明式 stage、policy、knowledge、schema、receipt contract 和少量无法声明化的 authority function。当前 P0 切口已经落到 OPL `family_scheduler_replacement` contract / Runtime Manager projection，MAS 默认 scheduler façade 已迁到 OPL replacement；本轮进一步补齐 `functional-agent-runtime-harness`、`functional_privatization_audit` unified read model、`family-runtime-lifecycle-index` refs-only SQLite sidecar index、`generic-substrate-projection`、完整 `Agent Lab` 控制面、`private_functional_surface_admission_policy` 和 `domain-pack-compiler` generated-surface handoff read model，用构造状态机验证 queue/stage attempt、typed closeout、refs-only memory writeback、human gate、retry/dead-letter 和 repair route，并把 MAS/MAG/RCA 私有化功能模块按代码路径、active caller、迁移动作、保留理由和不能上收理由归类为 OPL hosted/generated surface、declarative pack、minimal authority function、refs-only adapter、diagnostic cleanup path 或 provenance/fixture。
- `验收顺序`：每个功能迁移先跑 focused replacement proof、direct/hosted parity、receipt refs 对账、no-forbidden-write、no-active-caller proof 和 functional harness coverage；cross-repo integration、provider SLO 和 live soak 是后置生产验收。
- `禁写口径`：OPL 可以声明 framework functional closure / live-soak-ready 时，仍不能声明 MAS paper closure、MAG grant closure、RCA visual closure 或 App/domain truth ownership 完成。

## 结论

当前 family 已经完成“目标形态的控制面骨架”：`OPL Framework -> One Person Lab App -> Foundry Agents` 三层产品认知成立，MAS/MAG/RCA 都能被 OPL 识别为 descriptor-level aligned 的 standard domain agent，三仓 stage plane、domain memory descriptor 和 privatized functional module audit 也都能被 OPL 只读解析。

MAS、MAG 与 RCA 理想目标态进一步校准了 family 边界：通用 runtime、queue、workspace/source intake shell、memory locator、artifact/package lifecycle、restore/retention、projection、workbench shell、route/decision visualization、review/repair transport、native-helper execution envelope、generic persistence / SQLite lifecycle index、observability primitive，以及 CLI/MCP/product-entry/sidecar/status 等外壳，应尽量由 OPL Framework / One Person Lab App 提供或生成；MAS/MAG/RCA 这类 Foundry Agents 应收窄成 `Declarative Domain Pack + minimal authority functions`，持有领域知识、路线/策略、质量 verdict、artifact/export authority、memory body/writeback decision、owner receipt、domain transition spec 和必要的 authority function。这个校准不改变当前 truth owner，只改变“后续缺口应该补在哪里”的判断：历史上已经手写在 domain repo 内的 thin adapter 不能被默认视为长期合理，它要么被 OPL pack compiler 生成，要么必须证明无法声明化且保留在 domain authority boundary。

这份计划不再以“当前仓库已经开发成这样”为前提来保护私有实现。当前实现是迁移约束和风险来源，不是目标架构。四个 repo 都可以为了目标态重构；只要某个现有功能面不属于标准 OPL Agent 的声明式 pack、refs-only boundary adapter 或 minimal authority function，就应进入功能/结构差距，并在计划中明确上收、替换、收薄或退役路径。测试/证据差距只用于记录目标结构已经落地但还缺真实 receipt、workspace proof、provider-hosted apply、App drilldown 或 long soak 的部分。
