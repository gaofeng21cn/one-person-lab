## 当前结论

OPL family 的目标形态已经明确：`OPL Framework` 是完整智能体开发/运行框架；`One Person Lab App` 是面向人的工作台；`MAS`、`MAG`、`RCA` 是基于 OPL 的标准 Foundry Agents。标准 Agent 目标形态是 `Declarative Domain Pack + OPL generated/hosted surfaces + standard authority functions`。

当前 OPL 已具备 framework 主干：domain descriptor / stage / action / memory discovery、Temporal-backed provider code、OPL-owned provider scheduler cadence/tick surface、typed family queue、stage attempt ledger、typed closeout、safe runtime action shell、lifecycle refs-only SQLite index、external evidence refs-only receipt ledger、App/operator drilldown read model、App safe-action execution bridge、production closeout read model、functional runtime harness、generic substrate projection、domain pack compiler active-caller proof、private functional audit read model、Agent Lab 控制面、checkout-clean command runner 和 standard agent conformance report。Temporal 是 production online runtime 的必需 provider；`local_sqlite` 只能作为 dev/CI/offline diagnostic baseline，不是生产 daemon 或 readiness fallback。`opl family-runtime status` 现在直接投影 `periodic_execution` / domain-daemon replacement summary，App drilldown 也通过 `periodic_execution_refs` 暴露 scheduler status/install/trigger/tick 与 provider SLO receipt refs，并通过 `app_execution_bridge` 声明 App 如何把 safe action route 提交到 `opl runtime action execute`。

2026-05-19 对“物理源码形态标准化”的 fresh 结论是：family 层已有标准 OPL Agent 目标形态和 conformance gate，但四个 agent repo 的完成口径必须继续区分 `semantic/contract standardization` 与 `physical source morphology standardization`。成熟框架经验也支持这种拆分：OpenAI Agents SDK 把 agent 配置与 runner/session/handoff/guardrail 分开，LangGraph 把 graph/persistence/checkpointer/store 分开，AutoGen 把 agent/team/workbench/state 分开，CrewAI 把 declarative agents/tasks/tools 与 crew/process 分开。OPL 吸收的是“声明包、authority tools、runtime orchestration、state persistence、workbench/evidence gate 分层”的模式，不引入外部 runtime dependency。

标准物理源码形态 gate 如下：

- `agent/` 必须是 canonical domain semantic pack，真实承载 stage、prompt、skill、knowledge、quality gate 和 policy；README、空目录或 `src/` / `packages/` 不能替代它。
- `contracts/` 承载 machine-readable descriptor、pack compiler input、stage/action/memory/artifact/receipt contract、generated handoff、evidence request 与 cleanup gate。
- `runtime/authority_functions/` 或等价 domain module 只保留 minimal authority function、native helper、receipt signer、fixtures；输出必须是 refs、receipt、typed blocker 或 safe action metadata。
- `src/` / `packages/` 中的 product-entry、sidecar、status、session、workbench、runtime、supervisor、scheduler、managed run 等代码必须逐项归类为 direct domain handler、refs-only adapter、authority function、diagnostic、history/tombstone 或 OPL generated/hosted surface target。
- 若 active source 命名仍让读者误以为 domain repo 持有 generic runtime、scheduler、session store、attempt ledger、workbench 或 generated wrapper owner，即使合同分类正确，也继续是 physical morphology cleanup tail。

2026-05-19 的系统评估把 OPL 自身语义治理收敛为六大冗余/语义污染点的一次性闭环。当前 main 已把它们落成可机器验证的 framework guard surface；剩余工作不是继续补框架阶段计划，而是在这些守门面之上补 MAS/MAG/RCA、App 和真实 production caller evidence。六个污染点与已落地守门面如下：

- `provider_readiness_truth_drift`：`opl family-runtime status`、`opl status runtime`、`opl runtime manager` 和 App provider projection 必须消费同一个 lifecycle-aware provider readiness payload；不能一处读 managed lifecycle/projection，另一处只读 env，再由文档解释差异。
- `generated_surface_drift`：CLI/MCP/Skill/product-entry/sidecar/status/session/workbench/OpenAI tool/AI SDK tool 必须从 canonical action/stage metadata 与 pack compiler input 派生；`blocked`、`drift_detected` 或 domain generated owner claim 不能作为常态存在。
- `app_operator_drilldown_overprojection`：App/operator 默认面应 summary-first，按 owner/domain/ref kind lazy drilldown；巨量 refs/routes 只能在显式 full/detail 模式出现，避免让 App projection 变成第二 runtime。
- `family_runtime_cli_parser_monolith`：`family-runtime` 命令入口必须拆成 provider、service/worker、scheduler、attempt、queue、lifecycle、evidence 等自然 parser 模块；public API 保持稳定，内部不继续堆叠单个超长函数。
- `stage_launch_guarantee_ambiguity`：stage launch 不能只写成“可启动”；stage admission、selected executor binding、idempotency、source/artifact/workspace scope refs、runtime assumptions、monitor refs、expected receipt obligations 和 `guarantee_mode` 必须在 proof-bundle / control-plane / App projection 中可见。启动安全、越权、effect-boundary runtime event、组合、human gate 或 executor binding 不满足时 fail closed；cohort / monitor / runtime-budget 这类轻量增强缺口默认进入 readiness warning / drilldown，避免把 AI 执行路径写死。
- `legacy_vocabulary_active_leakage`：Hermes/Gateway/frontdoor/local-manager/MDS-default/default-compat 等旧词只留在 history/tombstone/provenance、fixture、negative guard 或 cleanup plan；active docs、status 和 App 默认面只讲当前 truth owner 与替代 surface。

这六项已经形成 OPL-framework 侧 runtime single truth、generated surface gate、App drilldown summary-first、family-runtime parser split、stage scope / `guarantee_mode` projection 和主文档 vocabulary hygiene。它们的长期价值是让现有 framework surface 稳定交给 App、MAS/MAG/RCA 和未来新 Agent 使用，而不是继续扩散相似读面；它们不证明 MAS/MAG/RCA production ready，也不把 domain truth、quality/export verdict 或 artifact authority 迁入 OPL。

GraphFlow / GFL 论文中可吸收的模式已经被定位为 OPL 自有 contract pattern，并按 AI-first、contract-light 原则降重：stage pack 先做 admission，stage 之间用 `requires` / `ensures` 组合，可信部分拆成 `verified_static_core`，AI / 人 / 外部系统 / domain verdict 等边界进入 `runtime_enforced_boundary`，再由 selected executor 启动已准入 stage pack。当前默认 operator 入口是 `opl stages readiness --domain <domain>`，它聚合 `family-stage-admission`、`family-stage-proof-bundle`、`runtime_event_refs` completeness、`family-stage-control-plane.runtime_assumptions / monitor_refs`、source/artifact/workspace scope refs、`guarantee_mode` projection、assumption lifecycle、stage pack registry / migration policy、replay certification 和 stage cohort-loop projection；这些 surface 只管启动安全、越权、关键边界结果记录、replay / audit / route-back，不引入 GraphFlow / GFL runtime dependency，也不把 GraphFlow 写成 OPL provider、planner、stage runner、executor、domain authority、quality verdict 或 completion guarantee。
