当前对“物理源码形态标准化”的结论是：family 层已有标准 OPL Agent 目标形态和 conformance gate，但四个 agent repo 的完成口径必须继续区分 `semantic/contract standardization` 与 `physical source morphology standardization`。成熟框架经验也支持这种拆分：OpenAI Agents SDK 把 agent 配置与 runner/session/handoff/guardrail 分开，LangGraph 把 graph/persistence/checkpointer/store 分开，AutoGen 把 agent/team/workbench/state 分开，CrewAI 把 declarative agents/tasks/tools 与 crew/process 分开。OPL 吸收的是“声明包、authority tools、runtime orchestration、state persistence、workbench/evidence gate 分层”的模式，不引入外部 runtime dependency。

标准物理源码形态 gate 如下：

- `agent/` 必须是 canonical domain semantic pack，真实承载 stage、prompt、skill、knowledge、quality gate 和 policy；README、空目录或 `src/` / `packages/` 不能替代它。
- `contracts/` 承载 machine-readable descriptor、pack compiler input、stage/action/memory/artifact/receipt contract、generated handoff、evidence request 与 cleanup gate。
- `runtime/authority_functions/` 或等价 domain module 只保留 minimal authority function、native helper、receipt signer、fixtures；输出必须是 refs、receipt、typed blocker 或 safe action metadata。
- `src/` / `packages/` 中的 product-entry、sidecar、status、session、workbench、runtime、supervisor、scheduler、managed run 等代码必须逐项归类为 direct domain handler、refs-only adapter、authority function、diagnostic、history/tombstone 或 OPL generated/hosted surface target。
- 若 active source 命名仍让读者误以为 domain repo 持有 generic runtime、scheduler、session store、attempt ledger、workbench 或 generated wrapper owner，即使合同分类正确，也继续是 physical morphology cleanup tail。

OPL 自身语义治理已收敛为 `opl system semantic-hygiene --json` 的机器 gate，并把 public Surface Budget 与 functional privatization evidence gate 上收进同一系统守门面。当前 main 的系统守门面保持 `production_ready_claim_count=0`、`domain_ready_claim_count=0`、`artifact_authority_claim_count=0`；剩余工作不是继续补框架阶段计划，而是在这些守门面之上补 MAS/MAG/RCA、App 和真实 production evidence。当前机器 gate 如下：

- `provider_readiness_single_truth`：Temporal-backed provider readiness 是 production substrate truth；local/offline provider status 不能替代 Full online readiness。
- `generated_surface_drift_owner_claim`：OPL 只持 generated wrapper 与 drift manifest，domain repository 继续持 truth、quality verdict、memory body 和 artifact authority。
- `app_operator_drilldown_overprojection`：App/operator drilldown 是 refs-only projection，不能授权 domain readiness、export quality 或 artifact mutation。
- `evidence_envelope_single_semantics`：stage evidence、external evidence、domain dispatch 和 cleanup receipt 共用 owner/scope/payload/claim/receipt/typed-blocker 读法，但不变成第二 readiness source。
- `public_surface_budget_conformance`：active public surface 必须有 allowed reason、promotion evidence、重复 consumer refs 与 false authority flags，避免默认 navigation surface 变成 AI planning 或 domain owner 的替代物。
- `functional_privatization_evidence_gate`：functional privatization 分类、默认 watchlist 清零或 compact read model 不能关闭 semantic-equivalence review / active private residue；必须有 domain/App/live evidence refs、typed blocker 或 owner receipt。
- `app_release_evidence_not_contract_only`：App release 需要真实 bundle、sidecar、截图、first-run/settings smoke 和 release verification；contract-only proof 不能替代用户路径证据。
- `family_runtime_parser_monolith`：`family-runtime` public API 保持稳定，内部按 provider / queue / lifecycle / stage 等职责拆分，不能把 parser 重新堆成语义 owner。

- `legacy_vocabulary_active_leakage`：旧 vocabulary 只能留在 history、diagnostic、provenance、fixture、negative guard 或 cleanup plan；active CLI、machine contracts 和 current docs 不恢复旧 truth alias。
