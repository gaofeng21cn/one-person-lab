# OPL 文档治理规则

## 目的

这份文档冻结 `OPL` 的文档管理方式，避免 docs 目录继续无边界膨胀，并保证 AI / 维护者核心工作集与公开四层文档体系各司其职。

## 一、核心工作集

以下五个文件是 AI / 维护者的默认工作集：

- `docs/project.md`
- `docs/status.md`
- `docs/architecture.md`
- `docs/invariants.md`
- `docs/decisions.md`

它们负责回答项目目标、当前状态、边界和关键决策，不承担公开叙事的完整展示任务。

## 二、四层公开文档体系

### 1. 默认公开主线

回答 `OPL` 是什么、当前主线在哪里、读者应该先理解什么。
这一层必须双语同步、面向外部读者可读。

### 2. 公开合同与 Gateway 配套

回答 gateway 语义、shared foundation 边界、正式 contract surface。
这一层同样双语，但定位是公开技术合同，而不是首页叙事。

### 3. 参考级配套

承载 acceptance、example corpus、matrix、backlog、catalog、operating-surface 等参考材料。
可以继续 repo-tracked，也可以继续被引用，但不能反向定义主线。

### 4. 历史 specs / plans

`docs/specs/**` 与 `docs/plans/**` 只保留设计背景、执行计划和冻结过程，不参与当前主线竞争。

## 三、新文档落点判断

新增文档前，依次判断：

1. 它是核心工作集、公开主线、公开合同、参考级，还是历史记录。
2. 该角色是否已经有现成文档可复用。
3. 它是否真的需要进入默认公开面。

如果答案不清楚，默认下沉，不默认提升。

## 四、语言规则

- 对外公开文档必须双语。
- 内部技术、维护、参考、规划和历史文档默认中文。
- 允许保留英文术语、文件名、命令名、schema 和代码标识符。
- 避免无意义的中英混写。

## 五、历史与降噪

- `docs/history/omx/` 只保留历史入口，不再承担活跃 workflow。
- 优先通过索引和角色分层降噪，而不是把所有文件都继续摊在 docs 根目录。
