# OPL 文档治理规则

## 目的

这份文档用于冻结 `OPL` 仓库的文档管理方式，避免 docs 目录继续无边界膨胀，并防止参考材料反向改写公开主线。

## 一、四层分层规则

### 1. 默认公开主线

这层负责回答：

- `OPL` 是什么
- 当前主线处在什么阶段
- 读者应先理解哪些稳定边界

这层文档必须：

- 面向人类专家可读
- 与根 `README*` 的语气一致
- 中英双语同步
- 保持节制，不把技术细节堆成目录墙

### 2. 公开合同与 Gateway 配套文档

这层负责冻结：

- gateway 语义
- shared-foundation 边界
- substrate 与 domain 关系
- discovery / routed action / onboarding 这类正式合同面

这层文档也保持双语，但定位是“公开技术合同”，不是默认首页叙事。

### 3. 参考级配套文档

这层只承担：

- acceptance
- example corpus
- backlog
- rollout
- catalog
- matrix
- operating-surface 参考说明

它们可以继续仓库跟踪，也可以继续被引用，但必须被明确视为 `reference-grade`。
它们不应被根 `README*` 当作默认阅读路径，也不应被表述成当前治理主线。
面向 `OMX` 的长线提示词模板库也属于这一层：它服务执行接力与复制使用，但不成为仓库当前 phase / tranche 的 truth owner。

### 4. 历史 specs / plans

`docs/specs/**` 与 `docs/plans/**` 只用于保留设计背景、执行计划和冻结过程。
它们不是当前仓库的活文档真相面，不参与公开叙事竞争。

## 二、增删改文档时的判断顺序

新增文档前，先判断三件事：

1. 这份文档属于哪一层
2. 这一层是否已经有更合适的现有文档
3. 它是否真的需要进入默认公开面

如果答案不清楚，默认下沉一层，而不是默认提升一层。

## 三、公开面控制规则

- 根 `README*` 只链接默认公开主线，以及少量必须直达的合同文档。
- `docs/README*` 负责做 docs 总索引，不再把所有文件平铺到根 `README*`。
- 参考级文档允许被 `docs/README*` 收录，但应明确说明其角色只是参考，而不是主线。

## 四、语言与措辞规则

- 对外公开文档：必须双语。
- 内部技术、规划、备忘、设计记录：默认中文。
- 允许保留英文术语、文件名、命令名、schema、代码标识符。
- 不允许无意义的中英夹叙夹议；能用完整中文说清楚的，不要写成半中半英。
- 中文文档优先使用专业、简洁、可审核的表达，而不是口语化解释。

## 五、降噪原则

- 优先通过索引分层降噪，而不是大规模搬运历史文档。
- 只有当文件名或路径本身已经严重误导读者时，才考虑重命名或迁移。
- 参考级文档可以继续保留，但不应再被包装成“现在必须先读”的主线入口。
