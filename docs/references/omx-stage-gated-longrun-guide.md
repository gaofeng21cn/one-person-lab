# OMX 长线执行门控指引

## 文档目的

这份文档用于把“四仓可以长线连续跑一些，但不能无门槛一路跑到 `P2`”这件事冻结成统一执行规则。

与这份规则配套的可复制提示词模板，见：

- [OMX 长线提示词模板库](./omx-longrun-prompt-playbook.md)

它回答三个问题：

1. `OMX` 在四仓里怎样才算“可连续推进”
2. 哪些阶段可以自动继续，哪些阶段必须停下回报
3. 每次阶段切换前后，最少要留下什么 durable 证据

## 一、统一原则

### 1. 允许连续推进，但必须阶段门控

`OMX` 可以在同一条长线里按 `P0 -> P1 -> P2` 顺序持续推进，但不允许把三段视为一次性授权。

统一要求是：

- 每完成一个阶段，必须先验收当前阶段
- 只有当前阶段已经收敛，才允许进入下一阶段
- 一旦出现跨仓冲突、合同改写、边界摇摆或测试基线失真，必须停下回报

### 2. `Codex Host` 仍保留阶段裁决权

`OMX` 负责长时间执行，不负责自行扩阶段。

因此：

- `OMX` 可以在既定门槛内继续推进
- 但不能把“当前实现方便继续做”当成“当前应该继续做”
- 凡是会改写统一合同、扩能力面、改 formal entry 语义、动共享框架边界的动作，都不属于默认自动推进范围

### 3. 先收敛真相，再扩能力面

如果当前阶段的真相面还不稳定，后续阶段一律不自动启动。

这里的真相面至少包括：

- `README*`
- `contracts/project-truth/AGENTS.md`
- `CURRENT_PROGRAM.md`
- `LATEST_STATUS.md`
- `OPEN_ISSUES.md`
- 当前测试 / 验证 gate
- 当前 formal entry 说明

## 二、统一阶段模型

### 阶段 A：预检

进入长线前，`OMX` 必须先完成：

1. 读取仓内 `AGENTS.md`、`project-truth`、当前主线文档与 OPL 顶层合同
2. 确认当前唯一 active mainline
3. 确认本轮允许推进的最高阶段
4. 确认当前验证命令、报告路径、回写路径

若预检阶段发现：

- 当前主线不唯一
- formal entry 说明互相冲突
- 测试 gate 与状态文档明显不一致
- 任务本身需要跨仓裁决

则不得直接开做，应先回报。

### 阶段 B：`P0`

`P0` 的统一性质是：

- 收敛当前阶段最核心的不一致
- 先修合同、句柄、truth surface、fail-closed 行为、测试基线
- 不扩新能力面

`P0` 完成后，必须同步：

- 代码
- 测试 / 验证
- `CURRENT_PROGRAM.md`
- `LATEST_STATUS.md`
- `OPEN_ISSUES.md`

并输出一份“是否达到 `P1` 门槛”的明确判断。

### 阶段 C：`P1`

只有当下面条件全部满足时，`OMX` 才可自动从 `P0` 进入 `P1`：

- `P0` 已通过当前正式验证
- 当前主线真相面已一致
- 没有新增跨仓冲突
- 没有新增 formal entry 歧义
- 没有把产品 runtime 和开发控制面重新混写

`P1` 的统一性质是：

- 补 formal surface
- 补 repo-durable handoff surface
- 补 canonical spec / operating model / artifact map
- 把“已经存在但没冻结”的能力与边界补成正式真相

`P1` 仍不默认允许：

- 启动新产品线
- 大范围架构重写
- 共享框架抽取
- 统一平台 runtime 前置

### 阶段 D：`P2`

`P2` 默认不自动开启。

只有在提示词中明确授权，且当前仓满足下面条件时，`OMX` 才能继续：

- `P0` 和 `P1` 都已完成
- 当前测试 / 验证面稳定
- 当前主线边界稳定
- `P2` 只涉及本仓局部收尾，不会改写统一合同或跨仓边界

凡是涉及下列任一情况，`P2` 必须先停下回报：

- 新 formal entry
- 新 runtime surface
- 新共享框架抽取
- 新跨域协议
- 会改变 OPL 顶层叙事的结构动作

## 三、四仓默认最高自动推进上限

### `one-person-lab`

- 默认可自动推进到 `P1`
- `P2` 需要明确人工批准

原因：

- `OPL` 当前是顶层合同与文档 owner
- 其 `P2` 往往意味着共享框架、统一平台 runtime 或更高层级抽象，不应默认自动展开

### `med-autoscience`

- 默认可自动推进到 `P1`
- `P2` 需要明确人工批准

原因：

- `P0/P1` 主要是 outer-loop、formal surface、handoff surface 收紧
- `P2` 已经进入更强的 repo-durable operating model 和 artifact map 体系化，容易扩大边界

### `redcube-ai`

- 默认先只授权到 `P0`
- 只有当 `P0` 完成后明确证明 truth surface 与测试基线已收敛，才可继续 `P1`
- `P2` 需要明确人工批准

原因：

- 当前最大风险是真相面漂移，而不是能力缺失
- 若 `P0` 未收住，后续 `P1/P2` 都会建立在失真主线上

### `med-autogrant`

- 默认可自动推进到 `P0`
- `P1` 需要在 `P0` 完成后依据结果再决定
- `P2` 不自动开启

原因：

- 当前仍处于 `baseline hardening`
- `P2` 已经接近扩 runtime 能力面，不适合默认长线放权

## 四、每个阶段的最小回写要求

每完成一个阶段，`OMX` 最少应留下以下 durable 证据：

1. 本阶段完成了什么
2. 本阶段未完成什么
3. 当前验证命令与结果
4. 当前剩余 blocker / risk
5. 是否达到下一阶段门槛

优先回写到：

- `.omx/context/CURRENT_PROGRAM.md`
- `.omx/reports/<program-id>/LATEST_STATUS.md`
- `.omx/reports/<program-id>/OPEN_ISSUES.md`
- 必要的实现 / 测试 / 文档真相面

若仓内没有完整 `.omx` durable surface，则至少在会话输出中给出同等结构化结论，并停止继续扩阶段。

## 五、统一停止条件

出现下列任一情况，`OMX` 必须停止自动推进并回报：

- 当前阶段验证失败，且失败会改变后续判断
- 当前主线真相不唯一
- 文档口径与实现状态冲突
- 需要跨仓裁决
- 需要引入新入口或新能力面
- 需要改写 OPL 顶层统一合同
- 需要做大范围结构重写

## 六、给提示词的写法建议

给 `OMX` 的提示词应明确写出：

1. 本轮允许的最高阶段
2. 当前阶段的完成判据
3. 允许自动继续的门槛
4. 必须停止回报的条件
5. 必须同步更新的 durable surfaces

建议默认采用这种句式：

> 你可以在同一条长线里连续推进，但必须按阶段门控执行。  
> 默认先做 `P0`。只有当 `P0` 完成、验证通过、真相面一致、且未引入新的跨仓或合同冲突时，才可继续 `P1`。  
> `P2` 默认不自动开启；若进入 `P2` 需要先停下回报。  
> 每完成一个阶段，都要同步更新 `CURRENT_PROGRAM / LATEST_STATUS / OPEN_ISSUES` 或同等级 durable surface，并明确写出是否达到下一阶段门槛。

## 七、当前推荐用法

按当前四仓状态，推荐这样使用：

- `one-person-lab`：长线做到 `P1`
- `med-autoscience`：长线做到 `P1`
- `redcube-ai`：先只放 `P0`，`P1` 需以 `P0` 收敛结果为门槛
- `med-autogrant`：先只放 `P0`

这已经是“尽量让它们连续跑一些”的合理上限。

再往上放权，收益不如风险大。
